#### Firehose robustness, specifically

`skeet-prune` already consumes the firehose the efficient way — Jetstream (not the raw `com.atproto.sync.subscribeRepos` CBOR firehose) via `jetstream-oxide`, with `JetstreamCompression::Zstd`, a server-side `wantedCollections=app.bsky.feed.post` filter, and a shuffled four-endpoint list for connect-time failover. The transport is already lean; the gap is *consumption robustness*, not mechanism. Two failure-mode defects remain, both in `firehose.rs` / `firehose_stage.rs`:

- **Reconnects silently lose data.** `JetstreamConfig.cursor` is never set (we build the config with `..Default::default()`), so every reconnection starts in live-tail. A silent connection takes up to `recv_timeout` (30s) to notice, then up to ~5s × endpoints to re-establish — and every post in that whole window is gone, unobserved. On a redeploy or a flaky network this quietly punches holes in the stream we never see.
- **The reconnect loop has no backoff.** On connect failure `firehose_stage::run` does `warn; continue` with zero delay, and `max_retries: 0` disables the library's own retry — so during a real outage the DIY loop retry-storms all four public Jetstream instances as fast as it can. That's exactly the "unnecessary load on the central servers" we don't want to be a source of.

Goal: survive disconnects without losing observed posts, and stay a polite client even when things are on fire — without touching the (already-good) transport. The pipeline is already idempotent — the store is keyed by content hash / `SkeetId` — so replaying a few seconds of duplicate events on reconnect is safe and collapses to the same rows. That idempotency is the precondition that makes cursor-based resumption viable here.

Apply the cursor change first, then the backoff: backoff deliberately lengthens the disconnect gap, so the cursor (which makes gaps lossless) needs to be in place *before* we make gaps longer on purpose.

#### Group 1: bobby.houseofmoran.io should fall back to older data when newer unavailable

It's possible there could be an outage of the backing pruner process for > 48h which means what is shown on the feed or on website could expire gradually. I think what I'd like to do here is add a fallback mechanism where, for preferred Order + Limit, it will try successively older lists if it doesn't find anything in the preferred one. So, here's how it would roughly work:
* Finds all feeds available and orders them by age limit (i.e. 48h before 7d before 1y)
* When looking for a preferred feed (e.g. `quality-48h`) if that feed is empty, then it will find the next oldest available feed with the same ordering, in this case `quality-7d`. If that's also empty or missing then it should go to next.

This way when there is some sort of outage it gracefully degrades to older data.

**Decisions**
* **Home for the fallback: `skeet-publish`.** It already owns `FeedSource`/`PublishedImagesSource`/`RedisFeedSource`, the `PublishedList`/`PublishedListCatalog` keys, and `connect` — the fallback is one more source built from those. `skeet-feed` just wires it; `skeet-appraise` keeps its own `AvailableFeeds` (its dropdown/`resolve`/`UnknownFeed` policy differs and isn't a fallback).
* **Discovery is per-request, via the catalog** (the same shape `skeet-appraise` already uses). Re-reading the catalog + re-checking emptiness on each call is what makes degradation *live*: an outage that empties the preferred list mid-run, or a newly-published older list, is picked up without a restart. Cost is one extra `SMEMBERS` per request on the happy path.
* **Fallback chain = same `Order`, window ≥ preferred, ascending by `Limit::window()`.** "Next oldest with the same ordering" (doc) ⇒ stay within the preferred `Order` (e.g. `quality-48h` → `quality-7d` → `quality-4w` → `quality-1y`), never fall back to a *newer* window, never cross `Order` (quality→recency is out of scope). A missing preferred just means the chain starts at the next-oldest available list. "Empty or missing" is a single trigger because `PublishedList::read` already returns `[]` for both.

#### Group 2: cursor-based resumption

- Track `info.time_us` of the last processed event; set `JetstreamConfig.cursor` (a `DateTime<Utc>`) on reconnect, rewound ~5s for gapless playback. Currently `None` ⇒ every reconnect live-tails and drops the gap.
- Safe to replay because the store is idempotent (keyed by content hash / `SkeetId`).
- **Decision: in-memory only — no persistence (chosen).** The cursor lives in `firehose_stage::run`'s stack and survives reconnects *within a process*; on process restart it resets to `None` (live-tail) and the redeploy/crash gap is **accepted as lost**. This keeps the change small (a few lines, no new storage / startup-read / cursor-staleness concerns). Persisting it (to close the restart gap) is explicitly **out of scope** here — the review's §10 "drain on SIGTERM" is the *other* half of restart-loss and also stays deferred to its own work.

**API facts (verified against `jetstream-oxide 0.1.2`):** `JetstreamConfig.cursor: Option<chrono::DateTime<Utc>>`, default `None`, serialised on connect via `cursor.timestamp_micros()`; "absent or future cursor ⇒ live-tail". Every `JetstreamEvent` variant carries an `EventInfo { time_us: u64, .. }` (Commit `Create`/`Update`/`Delete`, `Identity`, `Account` — `time_us` is `#[serde(flatten)]`ed in), so the cursor can advance on *every* received event, not only on candidates. `chrono::DateTime::<Utc>::from_timestamp_micros(i64)` is the inverse used to rebuild the cursor.

#### Group 3: reconnect backoff

- Add exponential backoff with jitter between reconnect *cycles* (one cycle = one pass over all shuffled endpoints), capped ~30–60s; reset after a connection stays up.
- Currently `warn; continue` with no delay, and `max_retries: 0` disables the library's retry — so an outage retry-storms all four instances.
- Keep the DIY loop (it gives endpoint rotation, which `max_retries` doesn't); just add the delay.

**Decision: use the existing `backon` dep (`= "1.6"`, already workspace-level, used by `skeet-feed`'s redis reads) — don't hand-roll the schedule.** `backon::ExponentialBuilder` + the async `Retryable` extension (`(|| async {…}).retry(builder).notify(…).await`) gives jitter, capped delay, and unbounded retries off the shelf. `backon` wraps **our `firehose::connect`**, so one backon "attempt" *is* one cycle = one full pass over the shuffled endpoints — `connect` keeps rotating endpoints internally and `JetstreamConfig.max_retries` stays `0` (the DIY loop still owns rotation; `backon` only owns the *delay between cycles*). Composes with Group 2: each attempt recomputes `cursor_from(last_time_us)`, so even a long backoff resumes from the last-seen position. **API facts (verified against `backon 1.6.0`):** `ExponentialBuilder` has `with_jitter()`, `with_min_delay(Duration)`, `with_max_delay(Duration)`, `with_factor(f32)`, `without_max_times()` (unbounded — never give up); `.retry()`/`.notify(|err,&dur|…)`/`.when(pred)` hang off the `Retryable` trait; `RetryableWithContext` threads mutable state across attempts (the clean way to carry `last_time_us` if the `&mut` capture fights the borrow-checker).

#### Group 4: performance optimisations (in order of priority)

* [ ] **Parallelise the meta stage — the confirmed #1 bottleneck.** `prune_meta_stage::run` is a *single task* doing one serial `getPostThread` HTTP round-trip per candidate, so it caps the whole pipeline at the firehose intake rate with zero headroom. Evidence (≈1h, 5 runs, `logs/pruner.log.2026-06-24`): `depth: firehose=16` (full) in ~90% of *every* interval across *all* runs, and meta throughput tracks firehose throughput to within a few % every interval — the textbook "this stage is the constraint" signature; the image stage has headroom (`image` depth ~0, its lower rate is just the ~20% metadata-rejected before it). Fix: give meta a small worker pool over `getPostThread` (a `--meta-workers` CLI param mirroring `--image-workers`, default ~4–8), sharing the input via the **same `async-channel` MPMC the image fan-out adopts** — so do the §3 swap first and reuse one pattern for both stages. It's network-I/O-bound ⇒ throughput scales ~linearly with workers until the next stage becomes the limiter. **Double payoff:** raw throughput *and* it un-pins the firehose buffer, so `last_time_us` tracks real-time → cursor replays stay ~5s → much less H6 flap cost. Verify the firehose depth comes off the cap via `just clippy` + a live run (compare the depth/throughput status lines before/after).
    * **Bigger channel buffers are NOT the fix.** Raising the `16` cap just lets more queue before backpressure; if meta is the steady-state limiter the lag (and jetstream's unbounded internal buffer) still grows. Only the worker pool adds throughput.
    * **Higher-leverage alternative (bigger change, has a trade-off): reorder the metadata check to *after* image classification.** Today we `getPostThread` 100% of candidates, but image classification rejects ~80% and only ~0.3–0.7% are ever saved — so gating metadata on the image-survivors instead would cut `getPostThread` calls ~100× and erase the bottleneck without parallelising at all. Output is unchanged (a metadata-blocked post still isn't saved — it just gets classified first). Trade-off: we'd download+ML some author/labeler-flagged content we currently skip pre-download (self-labelled adult is already filtered inline at `extract`, so it's only the `getPostThread`-only labels), and image bandwidth/CPU rises — but the image stage has headroom. Pick this **or** the worker pool; the pool is the smaller, safer change.
    * **Sequencing of this group:** §3 (shared MPMC) → this meta pool → then, once meta is unblocked and the image stage becomes the next limiter, §4 (`spawn_blocking`) lets you safely raise `--image-workers`.
* [ ] **Replace the `Arc<Mutex<mpsc::Receiver>>` image fan-out with `async-channel` MPMC (§3) — decided.** `prune_image_stage` wraps the receiver in `Arc<Mutex<…>>` (`:30`) so N workers can share it (`rx.lock().await.recv().await`, `:88`) — the classic "mpsc is single-consumer" workaround. Swap to `async-channel` (a real MPMC channel, new workspace dep): each worker holds a clone of the `Receiver`, the `Arc<Mutex>` and the lock-then-recv both disappear, and the **per-worker detector ownership stays intact** (each worker still owns its `FaceDetector`/`TextDetector`). **Chosen over** the `buffer_unordered` restructure (§9), which has no per-worker state and so would force the detectors to become shared `Arc` + `Sync` — a change reaching into `face-detection`/`text-detection`, not just plumbing. Behaviour-equivalent; verify throughput is unchanged with `just clippy` + `just test-no-docker` (the pipeline has no unit test today — a small one driving the fan-out would de-risk this; optional).
* [ ] **Move CPU-bound classification off the async worker thread (§4).** `classify_image` (face + skin + optional text inference, tens-to-hundreds of ms, no `await`) runs inline in the async task (`prune_image_stage.rs:99`), monopolising a runtime thread for its duration — fine at `--image-workers 2`, a starvation footgun if it's raised on a 4-vCPU node. Wrap the `classify_image` call in `tokio::task::spawn_blocking` (detectors are already moved into the worker, so `Send` holds; share via `Arc` if `spawn_blocking`'s `'static` bound requires it). This is the one genuine correctness-under-load item, not tidiness — keep the change minimal and confirm the worker still pulls the next item only after the blocking classify returns (preserving the bounded-concurrency contract). Verify `just clippy` + `just test-no-docker`.

**Symptom (observed 2026-06-23, local run on a residential connection).** After Groups 2/3 landed, the pruner flaps: it connects, runs a while, the WS drops (`firehose channel closed`), and the reconnect backoff escalates 1.7s → 3.8s → 5.2s → 14.6s → 18.9s → 54.4s as successive sessions shorten (54.8s, then 4.6s, 6.5s, 1.16s, 0.70s, 2.7s). Note up front: pre-Groups-2/3 the same drops were *silent and self-healing* (`warn; continue` → instant live-tail reconnect, no replay, no escalating delay) — so part of what's new is **visibility + amplification**, not necessarily more drops. Background analysis: `skeet-prune-review.md` §10 + the firehose research below.

**VERDICT (2026-06-24, trace run — `logs/pruner.log.2026-06-24` after `00:16:48`).** Root cause found: the WS is killed by a **decode/deserialize error inside `jetstream-oxide` that the library silently swallows** (`let _ = websocket_task(...)`), *not* a network drop or server close — see **H6**. The cursor (Group 2) turns one undecodable message into a *fatal replay loop* — which is what "I didn't used to see these" actually is: pre-cursor every reconnect live-tailed, so a decode-death lost a few seconds and self-healed. Status flips below: **H1 disproved** (no server close in trace), **H2 confirmed** (category resolved; exact payload still hidden), **H3 contraindicated** (reset-on-progress would *hammer* the loop), **H4 confirmed** (cursor is the amplifier; the **replay-age cap** is now the primary fix), **H5 confirmed contributor**.

#### H1 — The drops are Jetstream's normal server-side cycling (≈WS 1006), not a regression of ours — DISPROVED (2026-06-24)

The drops hit across *multiple* endpoints (east **and** west), and Bluesky documents this exact behaviour: [jetstream#27](https://github.com/bluesky-social/jetstream/issues/27) reports frequent **1006 (closed, no frame) every ~5–10s**; the [firehose docs](https://docs.bsky.app/docs/advanced-guides/firehose) say `bsky.network` connections "may drop… if your client implements auto-reconnection… impact should be minimal" — i.e. drops are *expected* and prompt reconnection is the designed remedy. If true, our job is to reconnect quickly + idempotently (the cursor already makes that safe), **not** to treat each drop as an outage to back off from (see H3).

* [x] Confirm the close code/reason per drop (needs H2). Expect **1006**; a *different* code (e.g. 1008/1009/1011, or a close frame with a reason) would point at us (bad cursor, message too big, policy) instead — that distinction decides whether H3/H4 are the whole story or there's a real client bug.

**Result (2026-06-24):** Disproved — there is **no server close at all**. With `jetstream_oxide=trace` the death shows *no* `Connection closed. The jetstream#27 1006 reports were a red herring for *our* symptom.

#### H2 — The real disconnect reason is being thrown away — CONFIRMED (suppressed); prerequisite for H1/H4

#### H3 — `STABLE_AFTER = 60s` exceeds the typical session length, so the backoff never resets → escalation spiral — REAL but the fix is now CONTRAINDICATED (2026-06-24)

`session_was_stable` only resets the backoff at ≥ 60s, but Jetstream cycles connections in *well under* a minute (H1). In the captured run the **first, healthy 54.8s session (~300 skeets processed) was classified a flap** (54.8 < 60), and every later session was shorter — so the backoff *only ever escalates*, to the 60s cap, leaving the pruner reconnecting ~once a minute and replaying ~a minute of backlog each time. Classifying a flap by **duration** is the wrong signal: "connected, received events, then the server cycled us" is normal and should reconnect *promptly*.

Tasks:
* [ ] Re-tune the reset signal from *duration* to *progress*: a session that received ≥1 event (made progress) resets the backoff regardless of length; reserve escalation for genuine connect-failures / sessions that delivered **zero** events. Thread an events-received count out of `run_session` and decide on that. (Pure seam stays unit-testable.)
* [ ] If keeping a duration component at all, drop `STABLE_AFTER` well below the observed session length (seconds, not 60s) so normal cycling doesn't read as flapping.
* [ ] Sanity-check the resulting behaviour against H1's data: a routine 1006 should now reconnect at ~base delay (≈1s + jitter), not climb to the cap.

**Result (2026-06-24): do NOT do the reset-on-progress fix.** With H6's poison-message loop, a cursored session *does* make progress (a replay burst) and then dies — so resetting the backoff on progress would reconnect immediately, replay straight back into the poison, and **hammer** in a tight loop. The escalating backoff is currently the *only* escape (it grows until the cursor ages out of the replay window and a reconnect live-tails past the bad message). `STABLE_AFTER=60s` being too high is still a real mis-tuning, but it's moot until H6's replay-age cap lands; revisit the duration/progress signal *after* the cap, not before.

#### H4 — Cursor replay amplifies under escalating backoff (and catch-up connections may be more fragile) — THEORY (needs H2 + an A/B)

Each reconnect cursor is `cursor_from(last_time_us)` — anchored to the last event seen, rewound 5s — so the replay window is *everything since the last event*, which grows with the backoff gap (a 54s backoff ⇒ ~54s+ replay burst). With H3's spiral this compounds: longer gap → bigger replay → pipeline (H5) falls further behind → `last_time_us` lags more. Separately, [jetstream#42](https://github.com/bluesky-social/jetstream/issues/42) notes catch-up (cursored) connections drop/reorder events and **`time_us` is non-monotonic during catch-up** — relevant to `last_time_us = Some(event_time_us(&event))`, which assumes monotonicity (the 5s rewind mostly absorbs it). The captured run is *consistent* with cursored reconnects dying faster than the initial live-tail one, but n is small.

Tasks:
* [ ] Add per-connect visibility: log the cursor and its replay span (`now − cursor`) at connect, and events-received + duration at each session end (feeds H3 too). Shows the replay growing with the backoff.
* [ ] **A/B isolation:** one run with the cursor forced to `None`. If session lengths stop shortening, catch-up fragility is real; if not, it's pure server cycling (H1) and the cursor is exonerated.
* [ ] *(Only if A/B implicates replay size)* cap the replay — e.g. if the gap exceeds some bound, fall back to live-tail (`None`) and accept that gap as lost, rather than replaying minutes. Decide against the Group 2 "accept restart gap as lost" precedent.

**Result (2026-06-24): CONFIRMED — the cursor is the amplifier.** The trace makes the A/B almost redundant: actual WS lifetimes shrink ~10s, ~10s, 0.8s, 0.2s, 0.2s as each cursored reconnect replays back into the same undecodable message (H6). The cursor converts an occasional, self-healing decode-death into a fatal loop. **The replay-age cap is promoted from "only if" to the primary fix** — it bounds the blast radius of *any* poison message regardless of its exact identity (tracked under H6). (`time_us` non-monotonicity from jetstream#42 remains a minor latent concern, absorbed by the 5s rewind.)

#### H5 — The meta stage is the throughput bottleneck, so the pipeline lags real-time → cursors lag → replays grow — CONFIRMED (from the run); contributor, not the drop cause

`depth: firehose=16` is pinned at the channel cap throughout, while `meta`/`image` depths stay low — i.e. the **single-threaded `prune_meta_stage`** (one serial `getPostThread` HTTP call per candidate) can't drain the firehose channel on residential latency. This does **not** drop the socket (jetstream-oxide buffers into `flume::unbounded`, so a slow consumer never stalls the WS read), but it keeps `last_time_us` behind real-time, enlarging every cursor replay (H4) and making it harder to ever "catch up". Review §3/§4 already flag the image stage; the meta stage has the same shape and is the current limiter.

* [ ] *(Separate task / its own change — not a quick fix)* parallelise the meta stage like the image stage (a small worker pool over `getPostThread`), so steady-state throughput exceeds the firehose rate and the pipeline tracks real-time. Gate on whether H3/H4 alone calm the flapping first.

**Result (2026-06-24): confirmed contributor, not the killer.** Still true that `depth: firehose=16` is pinned and the consumer lags ~tens of seconds (the `flume::unbounded` buffer also *hides* the real WS-death timing — there's a multi-second gap between jetstream-oxide's `retries exhausted` and our `firehose channel closed` as we drain the backlog, which is why "flapped after 34.7s" overstates the true ~10s/sub-second WS life). The lag enlarges the replay (feeding H4/H6) but doesn't cause the drop. Parallelising stays worthwhile (keeps the cursor's replay small in normal operation) but is secondary to H6's replay-age cap.

#### H6 — `jetstream-oxide` silently swallows a decode/deserialize error, tears the connection down, and leaks the ping task — CONFIRMED (2026-06-24); the actual root cause

At every death the trace shows `Client handshake done` → `Sending ping` / `Pong recieved` (connection genuinely alive) → `Connection retries exhausted`, with **nothing** between — no `Connection closed. Reason/Code` (trace on), no `Web socket error`, no `No web socket result`, no `All receivers dropped` (info on). The only path in `websocket_task` that returns *without logging any of those* is the `?` on a `JetstreamEventError` (zstd-decode or serde-deserialize into the atrium types), discarded by `let _ = websocket_task(...).await`. **Second, independent proof:** pings keep firing 30s apart from already-dead connections (conn #1 died `00:16:59`, its ping still fires at `00:17:19` and `00:18:19`) — the decode path returns *without* `ping_cancellation_token.cancel()` (only the socket-error/None branches cancel it), so every decode-death **leaks its ping task**. That leak is reachable only via the `?` path, which proves the exit branch. The exact offending payload/error text is **unrecoverable without forking** the library (it's thrown away); the trace nails the category, not the bytes.
